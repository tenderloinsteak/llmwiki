# MantisAlgo — UI Module Lint

## Overview

MantisAlgo's module registry (`config/module_registry/registry.json`) splits every module into a first-class `kind`: **ui** (chart display layer — what a buyer's screenshot looks like), **logic** (factory analysis/signal engines), **infra** (assembly plumbing). This skill is the read-only checker for the **ui** kind (26 modules as of 2026-07-17: `generators/families/ui/*.py` plus a handful of display-only modules living in other families, e.g. `struct_swing_labels`, `sig_bgcolor_zone`).

UI is the top sales lever for an indicator — buyers judge by screenshot before they read a line of code (`hermes/ui.md`). This skill exists because that standard used to be prose-only with no machine backing; `ui_audit.py` + `scripts/module_lint.py` turned it into something enforceable.

## When to Use

- 곽경준 says anything like "UI모듈 린트해줘", "UI 점검해줘", "UI 모듈들 확인해줘"
- Before running `scripts/factory_gate.py` or shipping/regenerating a SKU
- Right after adding or editing anything under `generators/families/ui/` or any module whose `registry.json` entry has `"kind": "ui"`
- As a periodic health check across the whole UI kind ("이 폴더 UI 다 깨끗해?")

Not for: adding a new UI module or changing an existing one's Pine code — that's `mantis-ui-module-update`. Not for logic/structure/momentum/volume modules — that's `mantis-strategy-module-lint`.

## How to Run

From the MantisAlgo repo root:

```bash
python3 scripts/module_lint.py --kind ui              # full report, every module
python3 scripts/module_lint.py --kind ui --quiet      # only modules with findings or advisories
python3 scripts/module_lint.py --module ui_signal_legend   # single module
python3 scripts/module_lint.py --kind ui --json       # machine-readable, for parsing
```

Exit code is `0` when there are zero **findings** (advisories don't fail the exit code — they're suggestions, not blockers).

## Reading the Output

Each module prints as `[ok]`, `[warn]` (advisories only), or `[FAIL]` (findings — real problems):

- **`group= 없는 input`** — every `input.*` must declare a `group=` so TradingView's settings panel is organized (core → visual → advanced, per `hermes/ui.md`)
- **`tooltip= 없는 input`** — every `input.*` must explain itself in Korean; the gate threshold is 90% coverage per module
- **`게이트 없는 bgcolor`** — a background wash that's on by default is the #1 "Christmas-tree chart" anti-pattern; it must sit behind a `show*`/`not compactMode`/`showTier*` toggle defaulting to off
- **`ui kind인데 시각화 호출·표시 제어 변수·토글 input 어느 것도 없음`** — a module tagged `kind: ui` that produces nothing visible and defines no display-control input is probably mis-classified or dead code
- Advisories worth reading even though they don't fail: **`팔레트 외 색상`** (colors outside the signature teal/orange/gray grammar — fine for accents, but too many dilutes brand consistency) and **`est_lines vs 실제`** (the registry's line estimate has drifted from what the generator actually emits — cosmetic, but useful for `MASTER_PLAN.md` bookkeeping)

## After Findings Are Fixed

1. Re-run the same lint command and confirm `findings=0`.
2. Run `python3 scripts/factory_gate.py` — module-level cleanliness doesn't guarantee the *assembled* SKU stays clean (multiple UI modules can each be fine alone but crowd a chart together). The gate's `[UI] SKU-XX: ...` lines are the assembled-level version of the same checks.
3. If a shipping SKU's visuals changed, regenerate the review artifact: `python3 scripts/visual_sku_review.py` (writes `output/sku_visual_review.html`).
4. If `${WIKI_PATH}` is reachable in this session, this is a good moment to mention it's stale if module descriptions/counts changed — but a pure lint pass with no code changes usually doesn't need a wiki sync (that belongs to the update skill).

## Common Pitfalls

- Don't hand-edit `wiki/modules/*.md` if you also have llmwiki open — those pages are auto-generated by `registry_to_wiki.py` and get overwritten.
- A module reporting `advisory: 스냅샷 출력이 비어 있음/주석뿐` isn't necessarily broken — some UI modules (`ui_display_profile`, `ui_compact_mode`) are context-dependent pointers that only emit code when a sibling module is absent. Read the generator source before treating this as a real gap.
- `--kind ui` only lints modules *individually*, generated with a minimal smoke idea — it can't catch interactions between two UI modules that only clash when assembled together in a specific SKU. For that, trust `factory_gate.py`'s assembled-level UI audit.
